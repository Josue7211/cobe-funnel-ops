# Roadmap: COBE First 90 Days

## Overview

This roadmap reframes `cobe-funnel-ops` around the first 90 days of the COBE role instead of generic product hardening. The sequence mirrors the job post directly: first ship the DM sprint funnel that makes money, then lock in call routing and no-show recovery, then automate post-sale onboarding, then give leadership a daily revenue operating view, and finally add the internal tool cadence plus reliability work that makes the system usable every day.

The outcome is not "a nice demo." The outcome is an operator system that already behaves like the first quarter of work after hiring.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): committed 90-day work
- Decimal phases (2.1, 3.1): urgent insertions if the interview or implementation reality forces them

- [x] **Phase 1: ManyChat To Stripe DM Sprint Funnel** - Turn inbound DM intent into tagged leads, checkout handoff, and Meta-ready purchase tracking
- [x] **Phase 2: GHL Call Routing And No-Show Recovery** - Build the consult-call pipeline, owner routing, and recovery automation loop
- [x] **Phase 3: Client Onboarding Autopilot** - Convert payment events into provisioning, asset delivery, and community access handoff
- [x] **Phase 4: Daily Revenue Command Center** - Give operators and leadership one revenue dashboard across queue, payments, recovery, and attribution
- [x] **Phase 5: Internal Tool Factory And Ops Hardening** - Add the monthly internal-tool pattern, SOP output, and reliability improvements that reduce recurring work

## Phase Details

### Phase 1: ManyChat To Stripe DM Sprint Funnel
**Goal**: Ship the low-ticket DM funnel that captures demand, classifies intent, drives checkout, and logs revenue-visible events.
**Depends on**: Nothing
**Requirements**: [DM-01, DM-02, PAY-01, CAPI-01]
**Success Criteria** (what must be TRUE):
  1. An inbound DM or form-style lead can be captured from the UI and converted into a durable lead record with tags, owner, and next action.
  2. Checkout handoff creates Stripe-visible state, queue movement, audit history, and delivery entries without hidden manual steps.
  3. Lead, InitiateCheckout, and Purchase payloads are visible with naming conventions and match-key readiness for Meta CAPI.
**Plans**: 4 plans

Plans:
- [ ] 01-01: Finish the DM intake surface for IG/ManyChat/Typeform-style entry paths and shared tagging logic
- [ ] 01-02: Harden checkout generation, purchase confirmation, and queue/timeline side effects
- [ ] 01-03: Add CAPI event naming, validation, and replay visibility for the full DM-to-purchase path
- [ ] 01-04: Add scenario verification so the whole DM sprint can be run as a working proof flow

### Phase 2: GHL Call Routing And No-Show Recovery
**Goal**: Build the consult-call system the posting asks for: routing, reminders, no-show detection, and recovery.
**Depends on**: Phase 1
**Requirements**: [GHL-01, GHL-02, REC-01, REC-02]
**Success Criteria** (what must be TRUE):
  1. Booking events move leads through consult, booked, no-show, and recovery states with visible owner assignment.
  2. No-show handling creates durable recovery work: reminders, rebook prompts, escalation, and delivery/audit proof.
  3. The UI makes call routing and recovery feel like a first-class system, not a sidecar demo path.
**Plans**: 4 plans

Plans:
- [ ] 02-01: Expand booking and owner-routing logic to mirror a real GHL pipeline
- [ ] 02-02: Build explicit no-show triggers, escalation timers, and recovery branch state
- [ ] 02-03: Surface closer assignment, call outcomes, and recovery metrics in the operator view
- [ ] 02-04: Add webhook and replay tests for booked, no-show, recovered, and rescheduled paths

### Phase 3: Client Onboarding Autopilot
**Goal**: Turn successful payments into immediate fulfillment work with assets, links, invites, and operator visibility.
**Depends on**: Phase 2
**Requirements**: [ONB-01, ONB-02, ONB-03, COMM-01]
**Success Criteria** (what must be TRUE):
  1. A payment-confirmed lead can trigger one-click onboarding provisioning from the live UI.
  2. Folder links, SOP links, Kajabi/Skool/Discord handoffs, and onboarding status are visible and retryable.
  3. Failed or partial onboarding is obvious to the operator and produces queue/audit work instead of silent drift.
**Plans**: 4 plans

Plans:
- [ ] 03-01: Expand onboarding records into a real checklist with statuses, links, and ownership
- [ ] 03-02: Wire Kajabi, Skool, and Discord handoff states into the provisioning and delivery system
- [ ] 03-03: Add retry, exception, and alert handling for incomplete onboarding runs
- [ ] 03-04: Add operator-facing proof that post-sale work was provisioned correctly

### Phase 4: Daily Revenue Command Center
**Goal**: Give COBE the daily operating dashboard called out in the posting: Stripe, Meta, GHL, and funnel visibility in one place.
**Depends on**: Phase 3
**Requirements**: [REV-01, REV-02, REV-03, ATTR-01]
**Success Criteria** (what must be TRUE):
  1. The dashboard shows influenced revenue, booked calls, no-show leakage, recovery wins, and pipeline pressure.
  2. Operators can read the health of Stripe, Meta CAPI, GHL, Slack, Sheets, and scraping connectors from the same surface.
  3. The view is useful for daily operations, not just a static proof screen.
**Plans**: 4 plans

Plans:
- [ ] 04-01: Build a real command-center summary with daily/weekly/monthly operator KPIs
- [ ] 04-02: Add attribution and event-health modules for Stripe, Meta CAPI, and GHL
- [ ] 04-03: Add reporting outputs for Slack and Sheets that match the dashboard state
- [ ] 04-04: Make the revenue dashboard presentable for leadership and interview walkthroughs

### Phase 5: Internal Tool Factory And Ops Hardening
**Goal**: Prove the "one internal tool per month" mindset and make the system reliable enough to run every day.
**Depends on**: Phase 4
**Requirements**: [TOOL-01, TOOL-02, SOP-01, REL-01, REL-02]
**Success Criteria** (what must be TRUE):
  1. The repo contains at least one obvious internal-tool pattern that removes recurring manual work for operators.
  2. SOP-grade documentation, queue visibility, audit trails, and replayable flows are good enough for handoff.
  3. Tests, sync behavior, and realtime updates are stable enough that the app feels like an operating tool, not a fragile demo.
**Plans**: 5 plans

Plans:
- [ ] 05-01: Add an internal-tool workspace for repetitive operator jobs such as proof sends or comment triage
- [ ] 05-02: Add SOP-oriented copy, naming standards, and operator handoff artifacts inside the product
- [ ] 05-03: Harden realtime, sync, and failure states for daily operator trust
- [ ] 05-04: Add smoke and scenario coverage for the 4 major business tracks
- [ ] 05-05: Package the product as a reusable operating system for month-two and month-three builds
