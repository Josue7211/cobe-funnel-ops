# Requirements: COBE First 90 Days

**Defined:** 2026-04-05
**Core Value:** An operator can run the exact first-90-day COBE funnel systems from one trustworthy internal console: DM to checkout, call routing and no-show recovery, post-sale onboarding, and daily revenue visibility.

## Phase 1 Requirements: ManyChat To Stripe DM Sprint Funnel

- [x] **DM-01**: Operator can capture inbound DM, comment-keyword, or form-style lead intake through one shared workflow entry path
- [x] **DM-02**: Lead intake applies durable tags, ownership, source attribution, and next action state in backend records
- [x] **PAY-01**: Checkout handoff can create Stripe-visible state and move the lead into the right queue/timeline path
- [x] **CAPI-01**: Lead, InitiateCheckout, and Purchase events are visible with Meta CAPI naming and match-key readiness

## Phase 2 Requirements: GHL Call Routing And No-Show Recovery

- [x] **GHL-01**: Booking events move leads through consult, booked, no-show, and recovery states with visible owner assignment
- [x] **GHL-02**: GHL-style routing logic is reflected in queue, timeline, audit, and operator workspace surfaces
- [x] **REC-01**: No-show handling creates durable recovery work including delivery, audit, and timeline proof
- [x] **REC-02**: Recovered and rescheduled outcomes are replayable and measurable from the operator console

## Phase 3 Requirements: Client Onboarding Autopilot

- [x] **ONB-01**: A payment-confirmed lead can trigger onboarding provisioning from the live UI without hidden manual steps
- [x] **ONB-02**: Folder links, SOP links, and onboarding status are visible and durable in backend state
- [x] **ONB-03**: Onboarding failures or partial runs surface clearly and support retry instead of silent drift
- [x] **COMM-01**: Kajabi, Skool, and Discord handoff states are visible as part of the onboarding workflow

## Phase 4 Requirements: Daily Revenue Command Center

- [x] **REV-01**: Dashboard shows daily operator metrics across revenue, booked calls, no-show leakage, recovery wins, and pipeline pressure
- [x] **REV-02**: Reporting reflects live backend state instead of static seed-only presentation
- [x] **REV-03**: Slack and Sheets outputs can be generated from the same live metrics shown in the dashboard
- [x] **ATTR-01**: Attribution and event-health views expose Stripe, Meta CAPI, and GHL status clearly enough for operator use

## Phase 5 Requirements: Internal Tool Factory And Ops Hardening

- [x] **TOOL-01**: The product includes at least one internal-tool workflow that removes a recurring operator task
- [x] **TOOL-02**: The system is structured so new internal tools can be added without breaking the main funnel surfaces
- [x] **SOP-01**: Naming, operator guidance, and handoff artifacts are clear enough for SOP-style reuse
- [x] **REL-01**: Realtime, sync, and failure handling are trustworthy enough for daily internal use
- [x] **REL-02**: Smoke and scenario coverage protect the four main business tracks from silent regressions

## Supporting Technical Requirements

- [x] **AUTH-01**: Operator can authenticate through the frontend and protected requests carry the correct session state
- [x] **AUTH-02**: Protected mutation endpoints reject unauthenticated access outside explicit local-dev allowances
- [x] **DEP-01**: There is one documented runtime where the operator UI and API work together outside localhost
- [x] **DEP-02**: CI runs backend smoke coverage in addition to lint and build
- [x] **DEP-03**: The live product path supports queue reads, workflow mutations, auth, and realtime without hidden local proxy assumptions
- [x] **SYNC-01**: Local SQLite state can push to and pull from Supabase without corrupting local runtime state
- [x] **SYNC-02**: Sync diff and reconcile flows provide usable visibility into local vs remote drift
- [x] **RT-01**: Realtime updates refresh affected operator views when workflow or sync events occur

## Out Of Scope

| Feature | Reason |
|---------|--------|
| Real Instagram authentication and production ManyChat ingestion | Not required to prove the operating system and would slow the 90-day build path |
| Full vendor parity with GHL, ManyChat, or Meta tools | The product should orchestrate work, not clone the vendor UIs |
| Consumer marketing pages or portfolio site work | Does not contribute to the internal operator product |
| Multi-tenant SaaS billing and account management | Outside the first-90-day internal tooling scope |
